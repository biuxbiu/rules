# Git 自动提交规则

## 目的

自动化 Git 提交流程，根据代码改动内容智能生成 commit message，并自动执行 add + commit + push。

## 触发方式

当用户提到以下关键词时应用此规则：

- `@gt`
- `提交代码`
- `提交到 git`
- `git 提交`
- `commit`
- `push 代码`

---

## 工作流程

### 1. 检查 Git 状态

首先执行 `git status` 查看当前改动：

- 已修改的文件
- 新增的文件
- 删除的文件
- 未跟踪的文件

### 2. 分析改动内容

根据 `git diff` 和 `git status` 的输出，分析改动类型：

**Emoji 映射规则：**

| Emoji | 类型     | 说明                       | 示例                        |
| ----- | -------- | -------------------------- | --------------------------- |
| ✨    | 新功能   | 添加新特性、新组件、新页面 | `✨ 添加用户登录功能`       |
| 🐛    | Bug修复  | 修复问题、错误             | `🐛 修复登录按钮点击无响应` |
| 📝    | 文档     | 更新文档、注释、README     | `📝 更新 API 文档`          |
| 🎨    | 样式     | UI/样式调整、格式化代码    | `🎨 优化按钮样式`           |
| ♻️    | 重构     | 代码重构、优化结构         | `♻️ 重构用户模块代码`       |
| ⚡️    | 性能     | 性能优化                   | `⚡️ 优化图片加载速度`       |
| 🔧    | 配置     | 修改配置文件               | `🔧 更新 webpack 配置`      |
| 🚀    | 部署     | 部署相关                   | `🚀 部署到生产环境`         |
| 🗑️    | 删除     | 删除文件、代码             | `🗑️ 删除废弃的组件`         |
| 🔀    | 合并     | 合并分支                   | `🔀 合并 dev 分支`          |
| 🎉    | 初始化   | 项目初始化                 | `🎉 初始化项目`             |
| 🔒    | 安全     | 安全相关修复               | `🔒 修复 XSS 漏洞`          |
| ⬆️    | 依赖     | 升级依赖                   | `⬆️ 升级 React 到 18.3`     |
| 📦    | 构建     | 构建相关、打包             | `📦 优化打包体积`           |
| 🌐    | 国际化   | 多语言支持                 | `🌐 添加英文翻译`           |
| ♿️    | 可访问性 | 无障碍优化                 | `♿️ 添加键盘导航支持`       |
| 💄    | UI       | UI 组件更新                | `💄 更新按钮组件样式`       |
| 🚧    | WIP      | 进行中的工作               | `🚧 用户中心开发中`         |

### 3. 生成 Commit Message

**格式规范：**

```
<emoji> <简短描述>
```

**描述规范：**

- 使用中文
- 简洁明了（建议 10-30 字）
- 描述做了什么，而不是为什么
- 使用动词开头：添加、修复、更新、优化、删除、重构

**多文件改动时的策略：**

1. **同类型改动**：使用统一 emoji

   ```
   ✨ 添加 Icon 组件和音乐播放器
   ```

2. **多类型改动**：选择主要改动类型

   ```
   🐛 修复音乐播放器 bug 并优化样式
   ```

3. **复杂改动**：拆分多次提交（建议）

   ```
   # 先提交功能
   ✨ 添加音乐播放器组件

   # 再提交样式
   🎨 优化音乐播放器样式
   ```

### 4. 执行提交流程

**步骤：**

```bash
# 1. 添加所有改动
git add .

# 2. 提交（使用生成的 message）
git commit -m "<emoji> <描述>"

# 3. 推送到当前分支
git push origin <当前分支>
```

**注意事项：**

- 在对话中不需要展示任务列表和步骤，只需输出 commit message。
- commit message 生成后，自动提交到当前分支。
- 每次 git push 之前先执行 `git pull`，需要拉取远程最新代码再提交，避免冲突。
- 一定要记得 git push 代码。（非常重要）
- 一定要记得 git push 代码。（非常重要）
- 一定要记得 git push 代码。（非常重要）

### 5. 特殊情况处理

**场景 A：没有改动**

```
提示：当前没有需要提交的改动
```

**场景 B：有未跟踪的文件**

```
询问：发现未跟踪的文件，是否要添加并提交？
- 列出文件清单
- 等待用户确认
```

**场景 C：有冲突**

```
提示：存在合并冲突，请先解决冲突后再提交
```

**场景 D：需要 pull**

```
执行：git pull --rebase
然后：重新执行 push
```

## 最佳实践

1. **原子提交**：每次提交只做一件事
2. **及时提交**：完成一个小功能就提交，不要积累太多改动
3. **清晰描述**：让别人（和未来的自己）能快速理解改动
4. **避免大规模提交**：如果改动太多，考虑拆分成多次提交

## 安全检查

在执行提交前，自动检查：

- ✅ 是否包含敏感信息（密码、密钥、token）
- ✅ 是否包含 `node_modules` 等不应提交的文件
- ✅ 是否在正确的分支上
- ✅ 是否有语法错误或 lint 错误

如发现问题，先提示用户，等待确认后再继续。
